## Principles

<AccordionGroup>
  <Accordion title="I. Narrow MVP scope" icon="target">
    The MVP serves three jobs only: record songs the user knows, capture songs
    the user is learning, and share that repertoire with a small circle.
    Everything else is out of scope until an amendment moves it in. **YAGNI is
    the default answer.**
  </Accordion>
  <Accordion title="II. Incremental delivery" icon="layers">
    Work ships in thin, demonstrable slices in this order: frontend → backend
    → LocalStack → AWS infrastructure (Terraform) → GitHub Actions
    deployment. Earlier stages must be usable, even with mocked data, before
    later ones begin.
  </Accordion>
  <Accordion title="III. Boring, proven stack" icon="anchor">
    React + TypeScript + Vite on the frontend. Python + `uv` + FastAPI +
    relational DB on the backend, with Clean / Hex / DDD applied
    pragmatically. AWS-native infra in Terraform. Docker + LocalStack for
    local validation. GitHub Actions for CI/CD. Mintlify for docs. New
    languages, frameworks, or providers require an amendment.
  </Accordion>
  <Accordion title="IV. Proportional rigor" icon="scale">
    Tests, accessibility, observability, and process show up when a real
    trigger justifies them: a recurring bug class, a regression, an incident,
    a compliance requirement, or logic complex enough that a test is the
    cheapest way to reason about it. "We might need it later" is not a
    trigger.
  </Accordion>
  <Accordion title="V. Docs-as-code, continuously" icon="file-text">
    Every user-facing or architectural change ships its Mintlify update in the
    same change set. If a change needs no doc update, that must be obvious
    from the change itself.
  </Accordion>
</AccordionGroup>

## Hard constraints

- Domain layer must not import frameworks, ORMs, or AWS SDKs.
- Secrets never committed. Local via `.env`; cloud via AWS-native stores.
- Relational DB by default. No NoSQL, queues, or caches without a feature
  that demonstrably requires them.
- Solo + AI default. The maintainer reviews every change.

## Amending

The maintainer may amend at any time. Amendment commits must:

1. Update the version line per the rules — MAJOR / MINOR / PATCH.
2. Update `Last Amended`.
3. Include a one-paragraph rationale in the commit message.
