## Overview

Campfire uses **Amazon Cognito** for identity management. Users can sign up and sign in with email/password, or continue with Google. Both paths converge on a single Campfire account per verified email address.

The full MVP flow is:

```
Landing → Sign up or Sign in → (Email verification for new accounts)
→ /me bootstrap → Onboarding (if required) → Home
```

Returning users with completed or deferred onboarding skip directly to Home.

## Authentication paths

### Email and password

1. User signs up at `/signup` with email and password.
2. Cognito sends a verification code; user confirms at `/signup` (confirmation step).
3. On first sign-in at `/signin`, `/me` is called, a Campfire local user record is created, and the user is routed to `/onboarding`.
4. Subsequent sign-ins return to `/app` directly if onboarding is complete or deferred.

### Google OAuth

1. User clicks **Continue with Google** at `/signin` or `/signup`.
2. Cognito Hosted UI initiates the OAuth redirect; the callback lands at `/auth/callback`.
3. `AuthCallbackPage` completes the OIDC flow via `completeRedirectSignIn()`, then calls `/me`.
4. If a Campfire account already exists for the same verified email, the Google identity is linked automatically — no duplicate account is created.
5. Routing follows the same onboarding-status rules as email/password.

### Password recovery

1. User clicks **Forgot password** on the sign-in page.
2. Cognito sends a reset code to the verified email.
3. User enters the code and a new password at the reset-confirm form.

## Session storage

Campfire stores a minimal session in `localStorage` under the key `campfire.session`:

```json
{
  "accessToken": "<Cognito JWT>",
  "displayName": "Ash Rivera",
  "email": "ash@example.com",
  "expiresAt": 1745000000000
}
```

Sessions are read by `getSession()` and invalidated if `expiresAt` is in the past. `signOut()` removes the session, dispatches `campfire-auth-changed`, and calls Cognito sign-out in non-dev environments.

## Identity linking

The backend resolves a Campfire user in priority order:

1. **Provider identity lookup** — find an existing `IDENTITY#<provider>#<subject>` record.
2. **Normalized email lookup** — find a user with the same lowercased, trimmed email.
3. **Create** — first visit; create a new `PROFILE` record and an `IDENTITY` link.

This guarantees a single Campfire account per verified email regardless of which provider was used first.

## Onboarding routing

`ProtectedRoute` wraps `/app` and `/onboarding` and loads `/me` to determine routing:

| `onboarding.status` | `/app` access | `/onboarding` access |
|---------------------|--------------|----------------------|
| `required`          | → `/onboarding` | ✓ allowed |
| `completed`         | ✓ allowed | ✓ allowed |
| `deferred`          | ✓ allowed | ✓ allowed |

## Preferences onboarding

`OnboardingPage` loads existing preferences via `GET /me/preferences` and shows the capture form:

- **Instruments**, **Genres**, **Play context**, **Goals**, **Experience level**
- Saving calls `PUT /me/preferences`, which persists preferences and marks onboarding `completed`.
- Skipping calls `PATCH /me/onboarding` with `{"action": "defer"}`, which marks onboarding `deferred`.

Both paths route to `/app` on success.

## Local development notes

- The local backend (`make run`) signs JWTs locally via `apps/api/src/main/local_auth.py`.
- Real Cognito flows require a dev Cognito user pool — see `infra/terraform/environments/dev/`.
- LocalStack provides DynamoDB and SSM locally; there is no Cognito emulator.
- End-to-end acceptance tests gated by `E2E_REAL_AUTH=true` require real Cognito credentials.
