---
title: "Register repertoire"
description: "Tie a (song, instrument, proficiency) triple to the current user."
---

## Endpoint

```http
POST /api/v1/repertoire
Content-Type: application/json
X-User-Id: <uuid>
```

## Request body

```json
{
  "song_title": "Black",
  "song_artist": "Pearl Jam",
  "instrument": "guitar",
  "proficiency": 8
}
```

<ParamField body="song_title" type="string" required>
  Non-empty.
</ParamField>
<ParamField body="song_artist" type="string" required>
  Non-empty.
</ParamField>
<ParamField body="instrument" type="string" required>
  Non-empty. Lowercased server-side.
</ParamField>
<ParamField body="proficiency" type="integer" required>
  0..10.
</ParamField>

## Response — `201 Created`

```json
{
  "entry_id": "…",
  "user_id": "…",
  "song_id": "…",
  "song_title": "Black",
  "song_artist": "Pearl Jam",
  "instrument": "guitar",
  "proficiency": 8,
  "proficiency_label": "advanced"
}
```

## Error map

| Status | Cause |
|---|---|
| `401` | Header missing or unknown user id |
| `409` | Duplicate `(user, song, instrument)` |
| `422` | Body validation failed (empty fields, proficiency out of range, wrong types) |

## Find-or-create song

The song is identified by `(title, artist)`. The use case calls
`find_by_title_and_artist` and creates a fresh `Song` if there is no match.
This keeps registration frictionless today; a canonical catalog can replace
this behavior without changing the endpoint contract.

## Uniqueness rule

A `RepertoireEntry` is unique per `(user_id, song_id, instrument)`. Same song,
different instrument is allowed (`Black` on guitar **and** `Black` on vocals).

<Warning>
  Proficiency is not versioned. Registering the same song + instrument twice
  returns `409` — it does not overwrite the previous score. An "update
  proficiency" endpoint is not part of the current slice.
</Warning>
