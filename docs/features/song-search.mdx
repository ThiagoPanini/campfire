title: "Song search"

---

## Endpoint

```http
GET /api/v1/songs/search?q=<text>&limit=<n>
```

- `q` — substring match on title **or** artist, case-insensitive.
- `limit` — 1..100, defaults to 20.
- Empty / whitespace `q` → `[]` (by design, this powers typeahead).

## Example

```bash
curl -H "X-User-Id: $USER_ID" \
  "http://localhost:8000/api/v1/songs/search?q=beatles"
```

```json
[
  {"title":"Hey Jude","artist":"The Beatles","source":"local","external_id":null},
  {"title":"Let It Be","artist":"The Beatles","source":"local","external_id":null},
  {"title":"Blackbird","artist":"The Beatles","source":"local","external_id":null},
  {"title":"Yesterday","artist":"The Beatles","source":"local","external_id":null}
]
```

## Source field

`source` identifies which provider served the result. The only provider today
is `local` — an in-memory seeded list inside
`infrastructure/persistence/memory/song_search_provider.py`. `external_id` is
reserved for future providers (MusicBrainz, Spotify, …) that surface their
own stable identifiers.

## Why search is a port

`SongSearchProvider` is a `Protocol` distinct from `SongRepository`. The
repository stores songs users have actually registered; the provider answers
"does this song exist somewhere I can offer you?". Swapping the provider to a
real external API touches `bootstrap.py` only.

<Tip>
  Known seeded titles you can use in examples: `Black` (Pearl Jam), `Blackbird`
  (The Beatles), plus the four Beatles titles above. Keep these if you trim
  the seed — the tests rely on them.
</Tip>
