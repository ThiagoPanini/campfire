Campfire is being built in thin, vertical slices. This page is the honest
inventory of what is missing, so nothing you encounter feels like a surprise.

## In scope today

- `X-User-Id` header → seeded user lookup (`PlaceholderAuthenticator`).
- Song search over a local seeded catalog (`SongSearchProvider` port).
- Curated instrument suggestions with optional substring filter.
- Registering `(song, instrument, proficiency 0–10)` entries for the current user.
- Listing the current user's repertoire, with derived proficiency label.

## Known limitations

<CardGroup cols={2}>
  <Card title="No durable storage" icon="database">
    Every repository is in-memory. Restarting the server wipes repertoire and
    song ids.
  </Card>
  <Card title="No real auth" icon="key">
    Anyone with Alice's or Bob's id can act as them. Do not deploy publicly.
  </Card>
  <Card title="No external music provider" icon="music">
    The seeded catalog is the only source. No MusicBrainz, no Spotify.
  </Card>
  <Card title="No frontend" icon="browser">
    The API contract is the product. Swagger is the only UI.
  </Card>
  <Card title="No admin flow" icon="user-gear">
    The authorized-user list is seeded at boot. There is no way to add a user
    through the API.
  </Card>
  <Card title="No mastery history" icon="chart-line">
    A proficiency score replaces the previous one. There is no timeline.
  </Card>
</CardGroup>

## Intentionally deferred

<AccordionGroup>
  <Accordion title="Group-level possible repertoire" icon="users">
    Answering "what can everyone present play tonight?" requires a presence
    model. The previous scaffold experimented with this; the current slice is
    deliberately user-centric. The seam returns through a new use case.
  </Accordion>
  <Accordion title="Durable persistence" icon="hard-drive">
    SQLite or Postgres adapters slot behind the existing repository Protocols.
    No domain change required.
  </Accordion>
  <Accordion title="Canonical song catalog" icon="book-open">
    Songs today are identified by `(title, artist)` and created on demand
    inside `RegisterRepertoireEntry`. A controlled catalog is additive.
  </Accordion>
  <Accordion title="Song metadata beyond proficiency" icon="tag">
    No key, version, capo position, tuning. Proficiency is the only signal.
  </Accordion>
</AccordionGroup>

## Assumptions baked into the current slice

1. Musician vs audience are **behavioral profiles**, not fixed roles.
2. A repertoire entry is unique per `(user, song, instrument)` — same song on
   a different instrument is allowed.
3. Proficiency is a single integer 0–10 per entry; the category label is
   derived, not stored.
4. Songs are identified by `(title, artist)` for now.
5. Instruments are lightweight normalized names — custom values accepted at
   registration time without a prior catalog entry.

<Tip>
  If you expect one of the deferred items and the API returns `404`, it is not
  a bug. Start from the [features overview](/features/overview) to see what is
  live.
</Tip>
