title: "Your first request"
description: "Guided five-step walkthrough that exercises the full current slice."
---

The server is running at `http://localhost:8000`. This page walks you through
the full loop end-to-end with curl. If you prefer clicks, do the same thing in
[Swagger UI](/test-locally/swagger-ui).

<Steps>
  <Step title="Health check">
    ```bash
    curl http://localhost:8000/api/v1/health
    ```

    ```json Expected
    {"status":"ok","version":"0.1.0"}
    ```

    If this is not `ok`, revisit [Run the backend](/get-started/run-backend).
  </Step>

  <Step title="Read Alice's id">
    Every authenticated route needs `X-User-Id`. To find Alice's id, open
    [Swagger](http://localhost:8000/docs) and execute `GET /api/v1/users` with
    any UUID — Swagger shows both seeded ids in the response body, even though
    the call returns `401`.

    <Tip>
      You can also start the server, then immediately open Swagger — the
      seeded ids are deterministic within a single process lifetime but
      regenerate on each restart.
    </Tip>

    Export the id for the rest of the walkthrough:

    <Tabs>
      <Tab title="Git Bash / macOS / Linux">
        ```bash
        export USER_ID="paste-alice-id-here"
        ```
      </Tab>
      <Tab title="PowerShell">
        ```powershell
        $env:USER_ID = "paste-alice-id-here"
        ```
      </Tab>
    </Tabs>

    Confirm:

    ```bash
    curl -H "X-User-Id: $USER_ID" http://localhost:8000/api/v1/users/me
    ```

    ```json Expected
    {"id":"…","email":"alice@example.com","display_name":"Alice"}
    ```
  </Step>

  <Step title="Search the seeded catalog">
    ```bash
    curl -H "X-User-Id: $USER_ID" \
      "http://localhost:8000/api/v1/songs/search?q=beatles"
    ```

    ```json Expected
    [
      {"title":"Hey Jude","artist":"The Beatles","source":"local","external_id":null},
      {"title":"Let It Be","artist":"The Beatles","source":"local","external_id":null},
      {"title":"Blackbird","artist":"The Beatles","source":"local","external_id":null},
      {"title":"Yesterday","artist":"The Beatles","source":"local","external_id":null}
    ]
    ```

    <Note>
      An empty `q` returns `[]` on purpose — the endpoint is built for typeahead.
    </Note>
  </Step>

  <Step title="Register a repertoire entry">
    ```bash
    curl -X POST http://localhost:8000/api/v1/repertoire \
      -H "X-User-Id: $USER_ID" \
      -H "Content-Type: application/json" \
      -d '{"song_title":"Black","song_artist":"Pearl Jam","instrument":"guitar","proficiency":8}'
    ```

    ```json Expected — 201 Created
    {
      "entry_id":"…","user_id":"…","song_id":"…",
      "song_title":"Black","song_artist":"Pearl Jam",
      "instrument":"guitar","proficiency":8,"proficiency_label":"advanced"
    }
    ```

    Send the same payload twice → `409 Conflict` (same `(user, song, instrument)` triple).
    Send it with `"instrument":"vocals"` → `201` again (different instrument = different entry).
  </Step>

  <Step title="Read your repertoire">
    ```bash
    curl -H "X-User-Id: $USER_ID" http://localhost:8000/api/v1/repertoire/me
    ```

    You get every entry you created in this process's lifetime, with the
    proficiency label computed on the fly.
  </Step>
</Steps>

<Check>
  You have now exercised the entire current slice — auth, search, write, read.
</Check>
