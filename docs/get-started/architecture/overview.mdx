<p className="cf-kicker">SYSTEM SHAPE</p>

Campfire ships in thin slices, in this order. Earlier stages must be usable
(even with mocked data) before later ones begin. This is a constitution rule —
see [Narrow MVP Scope & Incremental Delivery](/sdd/methodology/constitution).

| # | Stage | Status |
| --- | --- | --- |
| 1 | Frontend (mocked data) | <span className="cf-badge cf-badge--live">LIVE</span> |
| 2 | Backend identity slice | <span className="cf-badge cf-badge--partial">PARTIAL</span> |
| 3 | LocalStack-based local validation | <span className="cf-badge cf-badge--planned">PLANNED</span> |
| 4 | AWS infrastructure via Terraform | <span className="cf-badge cf-badge--planned">PLANNED</span> |
| 5 | GitHub Actions deployment pipeline | <span className="cf-badge cf-badge--planned">PLANNED</span> |

For the complete status matrix, use [Current status](/get-started/introduction/current-status).

## Repository layout

```text
campfire/
├── apps/
│   ├── web/             # React + TypeScript + Vite frontend
│   └── api/             # FastAPI identity backend
├── docs/                # Mintlify documentation site
├── specs/               # Spec-Kit features (spec, plan, tasks)
├── .specify/            # Constitution and Spec-Kit templates
├── .claude/             # Claude Code config, skills, agents
├── DESIGN.md            # Authoritative visual system
└── AGENTS.md            # Pointer for AI agents
```
