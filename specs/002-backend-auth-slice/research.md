# Phase 0 Research — Identity Slice

**Feature**: `002-backend-auth-slice`
**Date**: 2026-04-26
**Scope**: Justify each technology choice for the Campfire identity slice.
Each entry follows: **Decision → Rationale → Alternatives considered → Why
rejected**.

There are no NEEDS CLARIFICATION markers remaining. The clarifications log
in `spec.md` (session 2026-04-26) resolved every open question that touched
the contract.

---

## 1. Web framework — FastAPI

**Decision**: FastAPI `>=0.115,<0.116` on Python 3.12.

**Rationale**:
- The constitution (Principle III) names FastAPI explicitly. Re-litigating it
  for this slice would be a Principle III violation in itself.
- Native ASGI / async — required for `asyncpg`-backed request paths.
- First-class Pydantic v2 integration eliminates a serialization layer.
- Generated OpenAPI 3.1 schema is the spec's contract source of truth (per
  the prompt: "FastAPI's generated schema IS the contract").

**Alternatives considered**:
- **Litestar (formerly Starlite)**: equally async, also Pydantic v2-friendly,
  arguably nicer DI ergonomics. Rejected: not in the constitution's stack;
  smaller ecosystem; no upside for this slice's surface area (≈ 8 endpoints).
- **Starlette only (no FastAPI)**: lower-level, would force us to hand-roll
  Pydantic integration and OpenAPI generation. Rejected: rebuilds the parts
  of FastAPI we want for free.
- **Flask 3.x with `flask[async]`**: mature, familiar. Rejected: async story
  is a retrofit; OpenAPI requires extra plugins; constitution mandates
  FastAPI.

## 2. Validation & DTO layer — Pydantic v2

**Decision**: Pydantic `>=2.9,<3` for HTTP DTOs and `pydantic-settings` for
the settings provider.

**Rationale**:
- FastAPI integrates Pydantic v2 natively; no glue code.
- v2's Rust-backed core is ~5–10× faster than v1 — a concrete, observable
  win even at this slice's small request volume.
- `pydantic-settings` gives us a typed `Settings` model whose loader is the
  swappable seam we need for AWS Secrets Manager later.

**Alternatives considered**:
- **`attrs` + `cattrs`**: tighter, smaller. Rejected: no FastAPI integration;
  manual OpenAPI mapping.
- **`msgspec`**: faster than Pydantic v2. Rejected: FastAPI integration is
  experimental; ecosystem too thin for production paths we care about.

---

## 3. ORM — SQLAlchemy 2.x async

**Decision**: SQLAlchemy `>=2.0.36,<2.1` using the 2.x async API
(`AsyncEngine`, `AsyncSession`).

**Rationale**:
- Mature, battle-tested, the de-facto Python ORM.
- 2.x style aligns with type-checked declarative models (`Mapped[T]`),
  which makes the ORM↔domain mapping reviewable.
- Async support is first-class in 2.x; not a retrofit.
- Alembic integration is built-in.

**Alternatives considered**:
- **Tortoise ORM**: native async-first, simpler API. Rejected: smaller
  community, fewer escape hatches, weaker migration story (Aerich is less
  mature than Alembic).
- **SQLModel**: Pydantic-first wrapper over SQLAlchemy. Rejected: blurs the
  ORM↔DTO boundary, which is precisely the seam our hexagonal layering
  preserves; we *want* DTOs and ORM rows to be different types.
- **Raw `asyncpg` + hand-rolled SQL**: zero abstraction, fastest queries.
  Rejected: at this slice's complexity it's all downside (no migrations, no
  identity map, no schema-aware tooling). Re-evaluate when we hit a query
  the ORM can't express cleanly.

See ADR-002.

## 4. Postgres driver — `asyncpg`

**Decision**: `asyncpg >=0.30,<0.31` as the SQLAlchemy 2.x async driver.

**Rationale**:
- Fastest mainstream Postgres driver for Python.
- Pure-async — no thread-pool hop on each query.
- Compatible with serverless Postgres (Aurora Serverless v2) and RDS Proxy
  with appropriate pool sizing.

**Alternatives considered**:
- **`psycopg` (psycopg3) async**: also async-native, a closer match to
  psycopg2 ergonomics, supports both sync and async in one driver.
  Rejected: still measurably slower than `asyncpg`; no upside given we are
  greenfield (no legacy psycopg2 code to migrate).
- **`psycopg2` (sync)**: ruled out by the prompt — "asyncpg as the Postgres
  driver (NOT psycopg sync)".

See ADR-002.

---

## 5. Migrations — Alembic

**Decision**: Alembic `>=1.13,<1.14` with an async-aware `env.py`.

**Rationale**:
- Ships with SQLAlchemy; zero third-party glue.
- Hand-written migrations are the rule (autogenerate proposes, human
  reviews and commits). Rationale: identity tables are load-bearing — we
  cannot afford a missed `ON DELETE CASCADE` or a missed unique index.
- Idempotent seeds (`INSERT … ON CONFLICT DO NOTHING`) satisfy FR-021.

**Alternatives considered**:
- **`yoyo-migrations`**: simpler, plain-SQL files. Rejected: weaker tooling
  for autogenerated diffs (we still want them as a *suggestion* even if we
  don't trust them blindly), and no ORM integration.
- **Hand-rolled SQL files run by a script**: zero magic. Rejected: we'd be
  rebuilding the parts of Alembic that matter (revision graph, `down`/`up`
  transactions, env-aware DSN).

## 6. Password hashing — `argon2-cffi`

**Decision**: `argon2-cffi >=23.1,<24` (argon2id variant), with parameters
calibrated to the OWASP 2024 recommendation (memory ≥ 19 MiB, iterations
≥ 2, parallelism = 1) and re-tuned per environment.

**Rationale**:
- argon2id is the OWASP-preferred password-hashing algorithm — memory-hard,
  resistant to GPU/ASIC acceleration.
- The spec explicitly names argon2 (FR-001).
- `argon2-cffi` is the canonical Python binding, well-maintained, no
  surprising native-extension issues on Linux.

**Alternatives considered**:
- **`bcrypt`**: fine, widely deployed. Rejected: not memory-hard; OWASP and
  the spec both prefer argon2id for new systems.
- **`passlib`**: a meta-library that wraps argon2/bcrypt/scrypt. Rejected:
  abstraction we don't need; passlib's release cadence has slowed and the
  argon2-cffi dependency is what we'd pin anyway.
- **`pgcrypto` (Postgres-side hashing)**: works, keeps secrets out of app
  memory. Rejected: introduces a Postgres extension we'd need to verify on
  Aurora's allowlist, and ties the algorithm choice to the database
  engine (see ADR-004 — extensions are avoided in this slice).

## 7. JWT helper — `python-jose`

**Decision**: Keep `python-jose[cryptography] >=3.3,<4` as a declared
dependency; **do not** use it on the v1 happy path. Access and refresh
tokens are **opaque** server-issued strings (FR-007 / clarification
2026-04-26).

**Rationale**:
- The spec explicitly forbids JWT for v1 access tokens. Including the
  library now is anticipatory only — a follow-on slice that introduces
  federation (real Google OAuth, third-party id tokens) will need a JWT
  verifier, and we want the same dep across slices.
- python-jose is named in the prompt's stack; mentioned here for
  traceability.

**Alternatives considered**:
- **`PyJWT`**: leaner, more actively maintained. Rejected for now to honor
  the prompt's explicit choice; we may revisit when the federation slice
  arrives.
- **`authlib`**: full OAuth2/OIDC stack. Rejected: too broad for this
  slice; we'd carry a large surface area we don't use.
- **No JWT library at all in v1**: simpler. Rejected: we'd introduce a
  bigger diff in the federation slice when we add JWT verification — small
  cost to declare it now.

## 8. Engine — PostgreSQL 16

**Decision**: PostgreSQL 16 (`postgres:16-alpine`) for local dev; the
production engine is also Postgres-compatible. No specific managed offering
is selected in this slice (ADR-003).

**Rationale**:
- Constitution mandates a relational database. The spec's Assumptions
  pin Postgres-compatibility specifically (so the implementation does not
  adopt dialect features that block an Aurora migration).
- Postgres 16 is the current GA major as of 2026-04 and matches what AWS
  Aurora Serverless v2 supports (Aurora Postgres 16.x track is GA).
- Alpine image keeps Compose footprint tiny.

**Alternatives considered**:
- **SQLite**: simplest possible local story. Rejected — the spec's
  Assumptions explicitly rule it out, and we'd diverge from production at
  the boundary (e.g., `JSONB`, `timestamptz`, partial indexes).
- **MySQL / MariaDB**: also relational. Rejected by the same Postgres-
  compatibility constraint and by the constitution's preference for one
  engine.

See ADR-001 and ADR-003.

## 9. Integration test harness — Testcontainers (with Compose-DB fallback)

**Decision**: `testcontainers[postgres] >=4.8,<5` as the **preferred**
isolation mechanism (`postgres:16-alpine`, started per pytest session, schema
truncated between tests). **Fallback**: a dedicated `campfire_test` database
inside the dev Compose Postgres for environments where Testcontainers can't
launch a sibling container (notably WSL without Docker Desktop integration,
some hardened CI runners).

**Rationale**:
- Hermetic integration tests prevent the "works on my machine" failure
  mode and make CI a true reproduction of local.
- Real Postgres (not a mock, not SQLite) — the spec's intent is that
  integration tests catch dialect-level mistakes (FR-021 idempotency, the
  refresh-family revocation transaction, unique-constraint behaviour).
- Compose-DB fallback is documented in `quickstart.md` so contributors on
  WSL aren't blocked.

**Alternatives considered**:
- **Schema-per-test on a single shared DB**: faster for huge suites.
  Rejected: more glue code than Testcontainers' container reuse already
  gives us; harder to debug.
- **`pytest-postgresql`**: spawns a Postgres binary on the host. Rejected:
  requires `postgresql` to be installed on the host; doesn't match the
  Compose image; CI can't always install system packages.
- **Mocked DB / in-memory fakes for integration tests**: violates the
  prompt ("Integration tests MUST hit a real Postgres") and would let
  dialect bugs through.

## 10. Dependency manager — `uv`

**Decision**: `uv` as the dependency manager and project runner;
`pyproject.toml` is authoritative. Lockfile committed (`uv.lock`).

**Rationale**:
- Constitution names `uv` explicitly.
- Fastest resolver/installer in the Python ecosystem; cuts cold-install
  time in CI by an order of magnitude vs. pip-tools.
- Single tool for resolve / install / run, replacing
  pip+pip-tools+virtualenv stacks.

**Alternatives considered**:
- **Poetry**: mature, pyproject-native. Rejected: noticeably slower; less
  ergonomic for monorepo subdirectories (we'd be fighting its single-root
  assumption).
- **`pip-tools` + `venv`**: minimal surface, pip-only. Rejected: slower
  resolves, no scripts story, no lockfile transitivity guarantees on par
  with uv.
- **Hatch**: capable, with environments and scripts. Rejected: smaller
  ecosystem buy-in; constitution picked uv.

---

## 11. Async test runtime — `pytest-asyncio`

**Decision**: `pytest-asyncio >=0.24,<0.25` in `mode=auto` so async tests
work without explicit decorators.

**Rationale**:
- Standard, well-maintained, integrates with Testcontainers fixtures.
- `mode=auto` reduces boilerplate (every test in `tests/integration/` is
  async).

**Alternatives considered**:
- **`anyio` plugin (`pytest-anyio`)**: backend-agnostic (asyncio + trio).
  Rejected: we're asyncio-only (FastAPI/SQLAlchemy/asyncpg are all
  asyncio-native); the extra abstraction has no payoff.

---

## 12. Linter / formatter — `ruff`

**Decision**: `ruff >=0.7,<0.8` for both lint and format
(`ruff check` + `ruff format`).

**Rationale**:
- Single tool replaces flake8 + isort + black + most plugins.
- Fast enough that we can run it on every save without ceremony.
- `flake8-tidy-imports.banned-modules` codifies the architecture guard
  rails (routers must not import SQLAlchemy; domain must not import
  FastAPI; etc.) — this is the cheapest mechanism we have to enforce
  hexagonal layering.

**Alternatives considered**:
- **black + isort + flake8**: the legacy stack. Rejected: three tools
  where one suffices, and noticeably slower.

---

## 13. Type checking — `mypy` (advisory in this slice)

**Decision**: `mypy >=1.13,<2`, configured but **not** wired into CI as a
hard gate in this slice.

**Rationale**:
- Principle IV (Proportional Rigor): adopt rigor when justified. Type
  checking pays off most when refactors get bigger — this slice is small.
- Mypy is configured so contributors who want signal locally get it; CI
  promotion to a hard gate happens in the slice that introduces the second
  bounded context (when cross-context refactors first become real).

**Alternatives considered**:
- **Pyright**: faster, often more precise. Rejected: not a deal-breaker;
  mypy has the longer-running pyproject.toml integration.
- **No type checker at all**: rejected. Even advisory `mypy` catches
  enough at edit time to be worth the dependency cost.
