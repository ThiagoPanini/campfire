## Dependencies & Execution Order

### Phase Dependencies

- **Phase 1: Setup** can start immediately.
- **Phase 2: Foundational** depends on Phase 1 and blocks all protected flow work.
- **Phase 3: US1** depends on Phase 1 and can be validated before backend auth is complete.
- **Phase 4: US2** depends on Phase 2 and is required before US3 and US4 can be accepted.
- **Phase 5: US3** depends on US2 because onboarding requires authenticated user context.
- **Phase 6: US4** depends on US2 and US3 because returning-home behavior depends on auth and onboarding state.
- **Phase 7: US5** can begin after Phase 2 for audits, but final cleanup depends on US1-US4.
- **Phase 8: Polish** depends on the desired user stories being complete.

### User Story Dependencies

- **US1 (P1)**: Public entry; independent after setup.
- **US2 (P1)**: Auth core; depends on foundation.
- **US3 (P1)**: Onboarding/preferences; depends on US2.
- **US4 (P2)**: Returning home/logout; depends on US2 and US3.
- **US5 (P3)**: Scope hygiene/docs; depends on the implementation surface it reviews.

### Within Each User Story

- Write or extend tests first.
- Implement domain models before services.
- Implement services before HTTP handlers.
- Implement backend contracts before frontend integration.
- Validate each story independently before moving to the next priority story.

## Implementation Strategy

### MVP First

1. Complete Phase 1 and Phase 2.
2. Complete US1 so public entry remains stable.
3. Complete US2 so authentication is real and safe.
4. Complete US3 so new users reach onboarding and home.
5. Stop and validate the MVP path: landing -> sign-up/login -> verification or Google -> onboarding -> home -> logout.

### Incremental Delivery

1. Deliver US1 as public entry.
2. Deliver US2 as real authentication and single-email account safety.
3. Deliver US3 as onboarding and preferences persistence.
4. Deliver US4 as returning-user home and logout polish.
5. Deliver US5 and Polish as reviewability, docs, and cleanup.

### Suggested MVP Scope

The MVP scope is **US1 + US2 + US3**. US4 should follow immediately for a usable returning-user loop, and US5 should be included before merge if implementation removes or isolates existing code.

## Notes

- Do not count local mock auth or direct browser storage writes as MVP authentication acceptance.
- Keep credentials, reset codes, verification codes, Google secrets, and raw provider errors out of logs and docs.
- Preserve unrelated user/Claude work in the dirty worktree.
- Update docs in the same implementation slice as behavior changes.
- Commit only when the user explicitly asks or as part of a deliberate Spec-Kit hook invocation.
